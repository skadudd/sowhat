# Character System — 글쓰기 캐릭터 프로파일

sowhat의 논증을 산출물로 변환할 때 적용되는 **글쓰기 목소리** 시스템.
output-profiles(포맷: 무엇을 만드는가)와 독립된 레이어로,
**누가 쓰는가**(톤, 전개, 시그니처)를 정의한다.

```
논증 (Toulmin)  →  출력 형식 (output profile)  →  글쓰기 스타일 (character)
내용               포맷                          목소리
```

---

## 설계 원칙

1. **대표 문단이 핵심이다** — 규칙 나열이 아니라 레퍼런스에서 추출한 실제 문단이 캐릭터를 정의한다. Few-shot이 규칙보다 효과적이다.
2. **정량 수치를 쓰지 않는다** — "평균 문장 길이 14.2자"는 Claude가 실제로 계측할 수 없는 거짓 정밀도다. 대신 "짧은 문장 위주, 긴 문장은 드물게"처럼 정성적으로 기술한다.
3. **Positive only** — 단독 금지 예시는 프라이밍 위험이 있다. 대조쌍(정답+오답 함께 제시)은 허용 — 정답 쪽으로 끌리기 때문.
4. **2-pass 생성** — Pass 1(내용 정확성)과 Pass 2(캐릭터 적용)를 분리하여 관심사 충돌을 제거한다. 비용 +35~40%이나 금전적으로 무시 가능.
5. **3차원 분석** — 7차원은 과도하다. Voice(문장), Flow(문단), Persona(관계)로 축소한다.
6. **다층 캐릭터 정의** — 대표 문단(few-shot) + 문장 골격(구조 복제) + 어휘 팔레트(단어 선택) + 대조쌍(차이 인식) + 사고 습관(정보 배치). 각 층이 다른 레벨의 캐릭터 일관성을 담당한다.

## Character 파일 형식

### `character.md`

character.md는 **draft Pass 2에서 주입되는 파일**이다.
5개 층으로 캐릭터를 정의하며, 각 층이 다른 레벨의 일관성을 담당한다.

```
층 1: 대표 문단 (few-shot)     — "이런 느낌으로"
층 2: 문장 골격 (구조 복제)     — "이런 구조로"
층 3: 어휘 팔레트 (단어 선택)   — "이런 단어로"
층 4: 대조쌍 (차이 인식)        — "이것과 저것의 차이"
층 5: 사고 습관 (정보 배치)     — "이런 순서로 생각"
```

```markdown
---
name: my-voice
version: 3
created: 2026-03-26T14:00:00Z
updated: 2026-03-28T16:00:00Z

# Character: my-voice

이 글을 아래 캐릭터가 쓴 것처럼 다시 써라.
대표 문단의 톤을 흡수하고, 골격으로 구조를 잡고, 팔레트에서 단어를 골라라.

## 1. 대표 문단

### Voice — 이 사람의 문장

> "모두가 AI를 쓰고 있다고 말한다. 실제로는 쓰고 있는 척하고 있다.
> 도입 PoC는 100개가 넘지만 프로덕션에 올라간 건 세 손가락 안이다.
> 이유? 기술이 안 돼서가 아니다. 결정을 못 해서다."
— ref#1

> "거래가 절반 가까이 증발했다 (2023, 40% 감소).
> 남은 절반도 실수요가 아니었다."
— ref#3

### Flow — 이 사람의 전개

> "SaaS 대시보드를 만들면 PMF가 온다고 믿었다.
> 12개월, 3번의 피벗, 사용자 200명.
> 리텐션은 8%였다.
>
> PMF는 안 왔다. 정확히는, 올 수가 없었다.
> 만든 사람이 원하는 걸 만들었지 쓰는 사람이 원하는 걸 안 만들었으니까."
— cal#2

### Persona — 이 사람과 독자의 관계

> "3년간 매달 적자였다. 팀원은 7명에서 2명이 됐다.
> 그래도 피벗은 안 했다."
— ref#3

> "그러면 왜 아무도 이걸 안 했을까.
> 이유는 단순하다. 돈이 안 됐다."
— ref#4

## 2. 문장 골격

레퍼런스에서 추출한 재사용 가능한 문장 구조. 새 주제를 쓸 때 이 골격에 살을 붙여라.

- `[통념]. [반전].` — "모두가 X라고 한다. 실제로는 Y다."
- `[질문]? [부정]. [진짜 답].` — "이유? X 때문이 아니다. Y 때문이다."
- `[숫자 나열]. [숫자 나열]. [짧은 판정].` — "12개월, 3번의 피벗, 사용자 200명. 리텐션은 8%였다."
- `[해석] ([수치]).` — "거래가 절반 가까이 증발했다 (2023, 40% 감소)."
- `[전문용어] — [일상어 재설명].` — "CAC가 치솟았다 — 고객 한 명 데려오는 비용이 세 배가 됐다는 뜻이다."
- `[사실]. [사실]. [짧은 반전].` — "3년간 매달 적자였다. 팀원 7명에서 2명. 그래도 피벗은 안 했다."

## 3. 어휘 팔레트

이 캐릭터가 실제로 쓰는 표현. 새 글을 쓸 때 이 풀에서 골라라.

### 동사
- 감소: "증발했다", "빠졌다", "쪼그라들었다" (○) / "감소했다", "줄어들었다" (△)
- 증가: "치솟았다", "뛰었다", "불어났다" (○) / "증가했다", "상승했다" (△)
- 행위: "때려넣다", "갈아엎다", "들여다보다" (○) / "활용하다", "진행하다" (×)

### 연결
- 인과: "이유는 단순하다. ~다.", "~니까." (○) / "이는 ~에 기인한다" (×)
- 반전: "실제로는", "정확히는" (○) / "그러나", "하지만" (△ — 드물게)
- 부연: "— {풀어쓰기}" (○) / "즉,", "다시 말해" (△)

### 수량 표현
- "세 손가락 안", "절반 가까이" (○) / "약 3개", "약 50%" (△)

### 금기어
- "활용", "다양한", "혁신적인", "결론적으로", "종합하면"

## 4. 대조쌍

같은 내용의 두 버전. 이 캐릭터의 글과 일반적 글의 차이를 보여준다.

### 도입
이 사람: "모두가 AI를 쓰고 있다고 말한다. 실제로는 쓰고 있는 척하고 있다."
일반적: "최근 많은 기업들이 AI를 도입하고 있지만, 실제 프로덕션 적용 사례는 제한적입니다."
→ 차이: 통념("모두가~")→반전("실제로는") vs 일반화("최근 많은~")+접속사("하지만")

### 데이터
이 사람: "거래가 절반 가까이 증발했다 (2023, 40% 감소)"
일반적: "2023년 데이터에 따르면 거래량이 약 40% 감소하였습니다"
→ 차이: 해석 먼저("증발")+괄호 수치 vs "데이터에 따르면"+경어체

### 원인
이 사람: "이유? 기술이 안 돼서가 아니다. 결정을 못 해서다."
일반적: "이는 기술적 한계보다는 조직 내 의사결정 프로세스의 비효율성에 기인합니다."
→ 차이: 질문→부정→단문 답(9자) vs 한 문장에 모든 것을 담는 설명문

## 5. 사고 습관

이 캐릭터가 정보를 다루는 방식. 문장/어휘를 넘어 "생각의 순서"를 정의한다.

### 판단이 데이터보다 먼저
- "거래가 절반 가까이 증발했다" (판단) → "(2023, 40%)" (데이터)
- 일반적 순서의 역전: 데이터 → 해석이 아니라, 해석 → 데이터

### 원인을 한 문장으로 끊는다
- "결정을 못 해서다." (9자로 끝)
- 긴 분석/설명 대신 단정적 진단

### 감정을 팩트로 대체한다
- "힘들었다" 대신 숫자를 나열: "3년간 매달 적자. 팀원 7명→2명."
- 독자가 감정을 스스로 느끼게 유도

### 독자에게 생각할 틈을 준다
- 질문을 던지고 바로 답하지 않음. 1-2문장 뒤에 배치.
- "왜 아무도 이걸 안 했을까." (빈 줄) "이유는 단순하다."
```

**character.md 구조 요약**:
- 층 1 (대표 문단): ~800자 — 톤의 기준점 (few-shot)
- 층 2 (문장 골격): ~300자 — 구조 복제 템플릿

- 층 4 (대조쌍): ~400자 — 차이 인식 (positive 프라이밍)

- **합계: ~2100자** — 1회 주입 가능한 크기

### Phase 2: 대표 문단 선정 + 3차원 분석

각 레퍼런스에서 **대표 문단을 먼저 선정**한 뒤, 3차원으로 분석한다.

#### 2-1. 대표 문단 선정

레퍼런스 원문을 읽고, 다음 기준으로 대표 문단을 **3~5개** 선정한다:

- **도입 대표**: 글의 첫 1-3문단 중 가장 "이 사람답다"고 느껴지는 것
- **전개 대표**: 논거를 전개하는 중간부에서 가장 특징적인 문단
- **마무리 대표**: 마지막 1-3문단 중 가장 특징적인 것
- **증거 대표**: 데이터/수치를 다루는 문단 (있으면)
- **감정 대표**: 감정적 무게가 있는 문단 (있으면)

선정 시 **선정 이유**를 함께 기록한다.

#### 2-2. 3차원 분석 + 4개 층 추출

선정된 대표 문단을 기준으로 3차원 패턴과 4개 보조 층을 추출한다:

```
[Voice] — 문장 레벨
  이 사람의 문장은 어떤 특징이 있는가?
  - 문장 길이 경향 (짧은 편 / 긴 편 / 장단 교차)
  - 어휘 선택 (구어체/문어체, 한자어 처리, 전문용어 처리)
  - 종결 변주 패턴 (어떻게 문미를 다루는가)
  - 임팩트 장치 (초단문, 도치, 반복 등)

[Flow] — 문단 레벨
  이 사람의 글은 어떻게 흘러가는가?
  - 도입 패턴 (통념→반전? 질문? 훅?)
  - 문단 간 전환 방식 (접속사? 빈 줄? 첫 문장?)
  - 증거/데이터 삽입 방식 (인용? 문장 내? 괄호?)
  - 마무리 패턴 (결론? 열린 결말? CTA?)

[Persona] — 관계 레벨
  이 사람은 독자와 어떤 관계인가?
  - 독자 호칭 (여러분? 우리? 없음?)
  - 감정 처리 (직접? 사실 나열? 비유?)
  - 질문 사용 (수사적? 유도? 빈도?)
  - 거리감 (교수? 동료? 친구?)
```

3차원 분석과 동시에 **보조 층 4개**를 추출한다:

```
[문장 골격] — 대표 문단에서 재사용 가능한 구조 패턴 추출
  예: "[통념]. [반전].", "[질문]? [부정]. [진짜답]."
  레퍼런스 당 3~5개 골격 추출

[어휘 팔레트] — 이 레퍼런스에서 반복되는 특징적 표현 추출
  - 선호 동사, 연결 표현, 수량 표현
  - 레퍼런스에서 0회 사용된 범용어 → 금기어 후보

[대조쌍] — 가장 특징적인 문장을 일반적 표현으로 다시 써서 쌍 생성
  레퍼런스 당 1~2쌍 생성

[사고 습관] — 정보를 다루는 순서/방식에서 패턴 추출
  - 판단이 먼저? 데이터가 먼저?
  - 원인을 길게? 짧게?
  - 감정 처리 방식?
```

분석 결과는 정량 수치 없이, **패턴 설명 + 대표 문단 인용**으로 기록한다.

### Phase 3: 교차 검증

복수 레퍼런스의 대표 문단 + 패턴을 대조한다.

```
모든 레퍼런스에서 공통으로 나타나는 것:
  → character.md의 대표 문단 + 핵심 규칙으로 확정

일부에서만 나타나는 것:
  → 사용자에게 질문:
    "ref#1,2에서는 도입을 통념→반전으로 하지만, ref#3에서는 질문으로 시작합니다.
    [1] 통념→반전이 더 나다운 스타일
    [2] 질문 시작이 더 나다운 스타일
    [3] 둘 다 사용"

하나에서만 나타나는 것:
  → 해당 레퍼런스의 맥락적 선택으로 판단, 제외
```

교차 검증의 목적은 **character.md의 5개 층을 확정**하는 것이다:

- **대표 문단**: 9~15개 후보 → 최종 5~8개로 축소
- **문장 골격**: 레퍼런스 간 공통 구조 → 최종 5~8개 골격
- **어휘 팔레트**: 복수 레퍼런스에서 반복되는 표현 → 확정, 1회성 사용 → 제외
- **대조쌍**: 가장 특징적인 쌍 2~3개 선정
- **사고 습관**: 공통 정보 배치 패턴 → 3~4개 확정

## 캐릭터 성장 시스템

캐릭터는 정적 프로파일이 아니다. **콘텐츠를 만들수록 더 정교해지는 플라이휠**이다.

```
draft 생성 → 인간 audit → 피드백 → 캐릭터 업데이트 → 더 나은 draft → ...
         ↑                                                    |
         └────────────────────────────────────────────────────┘
```

### 성장 사이클

#### Cycle 1: 초기 빌드 (v1)

```
레퍼런스 3개 → 분석 → character.md v1
  대표 문단 5~8개, 골격 5~8개, 팔레트 초안, 대조쌍 2~3개, 사고 습관 3~4개
  → 캘리브레이션 (짧은+긴) 통과
  → 사용 가능한 상태. 하지만 아직 거칠다.
```

#### Cycle 2~N: Draft 피드백 루프

매 draft 생성 후, **인간 audit 단계에서 캐릭터 피드백이 자연스럽게 발생**한다.

```
/sowhat:draft 실행 → Pass 1 → Pass 2 → 산출물 생성

인간 audit:
  "전체적으로 좋은데, 3번째 문단에서 문장이 너무 길어졌어"
  "도입은 완벽한데 마무리가 너무 깔끔하게 정리됐어. 더 열려있어야 해"
  "여기 '활용하다'가 들어갔네"
  "'뛰었다' 대신 '폭발했다'가 더 나다워"
  "이 문단 너무 좋다. 이게 내 스타일이야"
```

이 피드백은 **3가지 유형**으로 분류된다:

### 피드백 유형

#### Type A: 교정 (Correction)
캐릭터가 깨진 곳을 지적. character.md를 **조정**.

```
피드백: "3번째 문단 문장이 너무 길다"

반영:
  → 핵심 규칙에 추가: "전개부에서 3문장 연속 긴 문장 금지"
  → 해당 문단을 수정하여 calibrations/에 before/after 저장
  → version +1
```

#### Type B: 발견 (Discovery)
인간이 "이건 좋다"라고 확인한 패턴. character.md를 **강화**.

```
피드백: "이 문단 너무 좋다. 이게 내 스타일이야"

반영:
  → 해당 문단을 대표 문단에 추가 (기존 약한 것 교체)
  → 해당 문단에서 새 골격 추출 가능하면 추가
  → version +1
```

이것이 핵심이다. **캘리브레이션 통과 샘플 + 인간이 확인한 실전 산출물**이 대표 문단에 합류하면, 캐릭터 정밀도가 매 사이클 올라간다.

#### Type C: 확장 (Expansion)
새로운 상황(주제, 포맷)에서 캐릭터가 어떻게 행동해야 하는지 학습.

```
피드백: "'뛰었다' 대신 '폭발했다'가 더 나다워"

반영:
  → 어휘 팔레트에 추가: 증가 → "폭발했다" (○)
  → 팔레트가 점점 풍부해짐 — 주제가 다양해질수록 어휘 범위가 넓어짐
  → version +1
```

### Draft 후 피드백 캡처 워크플로우

draft Step 7(검토) 이후:

```
산출물 검토가 끝났습니다. 캐릭터 피드백이 있나요?

[1] 캐릭터 피드백 있음 — 톤/스타일 관련
[2] 내용 수정만 — 캐릭터와 무관
[3] 완료

→ [1] 선택 시:

어떤 피드백인가요?

[1] 이 부분이 내 스타일이 아니다 (교정)
    → 어느 부분? 어떻게 달라야 하나?
[2] 이 부분이 특히 좋다 (발견)
    → 어느 부분?
[3] 표현/단어 관련 (확장)
    → 어떤 표현이 더 나다운가?
```

**복수 피드백 가능** — 한 draft에서 교정 2건 + 발견 1건 + 확장 1건이 나올 수 있다.

### 피드백 저장: `calibrations/cal-NNN.md`

```markdown
---
id: 5
type: draft-feedback           # build-calibration | draft-feedback
source_draft: "export/generated/linkedin-series/part-1.md"
version_tested: 3
feedbacks:
  - type: correction
    location: "para 3"
    detail: "문장이 너무 길다"
    action: "핵심 규칙에 전개부 긴 문장 제한 추가"
  - type: discovery
    location: "para 7-8"
    detail: "이 전개가 특히 좋다"
    action: "대표 문단 Flow에 추가 (기존 cal#2 샘플 교체)"
  - type: expansion
    detail: "'뛰었다' → '폭발했다'"
    action: "어휘 팔레트 증가 동사에 추가"
character_version_before: 3
character_version_after: 4
created: 2026-03-30T14:00:00Z
---

## 피드백 상세

### [correction] para 3
Before:
> "이 시장은 지난 3년간 매년 30%씩 성장하면서 동시에 주요 플레이어들의 시장 점유율이 재편되는 복잡한 양상을 보이고 있었다."

After (인간 수정 또는 재생성):
> "3년간 매년 30% 성장. 동시에 판이 바뀌고 있었다."

반영: 핵심 규칙 "전개부에서 30자 초과 문장은 의도적으로만"

### [discovery] para 7-8
> "PMF는 안 왔다. 정확히는, 올 수가 없었다.
> 만든 사람이 원하는 걸 만들었지 쓰는 사람이 원하는 걸 안 만들었으니까."

→ 대표 문단 Flow 섹션에 추가. 실전에서 검증된 앵커.

### [expansion] 어휘
"뛰었다" → "폭발했다" 추가
→ 어휘 팔레트 증가 동사: "치솟았다", "뛰었다", "불어났다", "폭발했다"
```

### 캐릭터 성숙도 모델

```
v1~v3: 초기 (Initial)
  레퍼런스 기반 추출. 대표 문단이 모두 레퍼런스에서 온 것.
  골격과 팔레트가 제한적.

v4~v8: 성장 (Growing)
  실전 draft 피드백 반영. 대표 문단에 캘리브레이션/실전 샘플 합류.
  어휘 팔레트가 확장. 새로운 주제에 대한 패턴 축적.

v9+: 성숙 (Mature)
  대부분의 draft에서 교정 피드백 없이 통과.
  대표 문단의 다수가 실전 검증 샘플.
  다양한 주제/포맷에서 일관된 캐릭터 유지.
```

```yaml
corrections_applied: 8
discoveries_added: 6
expansions_added: 15
---
```

## 2-pass 생성 + 붕괴 방지

### 핵심: 2-pass 생성

캐릭터 붕괴의 근본 원인은 Claude가 "내용 정확성"과 "캐릭터 유지"를 동시에 처리할 때 내용 쪽에 우선순위를 두기 때문이다. **관심사를 분리**하면 문제 자체가 줄어든다.

```
Pass 1 — 내용 생성 (캐릭터 없이)
  Input: 논증 내용 + output profile
  목표: 내용 정확성에만 집중
  Output: 중립 톤의 완성된 글

Pass 2 — 캐릭터 리라이트
  Input: Pass 1 결과 + character.md
  목표: 캐릭터에만 집중. 내용은 이미 있으므로 "설명 충동" 없음
  지시: "이 글을 아래 캐릭터가 쓴 것처럼 다시 써라."
  Output: 캐릭터가 적용된 최종 글
```

비용 증가: ~35~40% 토큰 (금전적으로 무시 가능).

### 보조: 사후 감사

Pass 2 완료 후:

```
[1] 금기어 검사 → grep으로 검색 (정확)
[2] 톤 비교 → 도입/중반/후반 각 1문단을 대표 문단과 비교

결과:
  통과 → 인간 audit으로
  금기어 발견 → 해당 문장 재작성
  톤 불일치 → 해당 구간 부분 재생성
```

## Draft 연동

### output-profiles.yml 확장

```yaml
# 기존 필드

deliverable: "linkedin-post"
# ...

# 신규 필드
character: "my-voice"          # 사용할 캐릭터 이름
                                # 미지정 시 캐릭터 없이 생성 (기존 동작)
```

### Draft 워크플로우

```
Step 1: 브리프 (기존)
Step 2: 캐릭터 선택 (신규)
  → 저장된 캐릭터 목록 + 성숙도 표시
  → 선택 또는 "캐릭터 없이 진행"
Step 3: 구조 (기존)
Step 4: Pass 1 — 내용 생성
  → 캐릭터 없이 중립 톤으로 생성
Step 5: Pass 2 — 캐릭터 리라이트
  → character.md 5개 층 적용
Step 6: 사후 감사
  → 금기어 + 톤 비교
Step 7: 인간 검토
  → 내용 검토 (기존)
Step 8: 캐릭터 피드백 (신규)
  → [1] 캐릭터 피드백 있음 → 유형별 캡처 → character.md 업데이트
  → [2] 내용 수정만
  → [3] 완료
```

캐릭터 미선택 시: Step 4만 실행, Step 5~6,8 건너뜀.

| output profile | ~500자 | — | 500자 |
| Pass 1 결과 | — | ~원문 크기 | (가변) |
| character.md | — | ~2100자 | 2100자 |
| **고정 비용** | | | **~5600자** |

## 신규 레퍼런스 추가 플로우

```
/sowhat:character
```
